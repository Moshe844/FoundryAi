# Foundry Engineering Guidance

Before planning, designing, or implementing a product change, read
[`docs/product-constitution.md`](docs/product-constitution.md). It is the
product's north star and the primary decision standard for this repository.

For every meaningful product decision, ask:

> Does this make Foundry feel like an elite engineering team, or does it make
> it feel like another AI tool?

Prefer decisions that strengthen project understanding, deliberate planning,
autonomous execution, continuous verification, context continuity, clear
professional reasoning, intentional design, project memory, and the complete
idea-to-production workflow.

Do not optimize a feature only for novelty, chat engagement, dashboard density,
or parity with AI coding tools. If a proposal does not make the case for choosing
Foundry over Claude, Codex, Cursor, Lovable, Bolt, Replit, or ChatGPT stronger,
redesign it or challenge whether it belongs.

In plans and handoffs, explain briefly how consequential product choices advance
the constitution. Treat this as product reasoning, not as a request to expose
private chain-of-thought.
