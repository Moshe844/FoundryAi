# Foundry Product Constitution

## The Promise

Foundry is not another AI chat, code generator, or IDE.

**Foundry is the place where software gets built.**

The user should feel like they hired an elite software team, not opened another
AI tool. Foundry should work beside them continuously, from an initial idea to a
production system and everything that follows.

## The Decision Standard

Judge the product by one question:

> If Claude, Codex, Cursor, Lovable, Bolt, Replit, and ChatGPT already exist,
> why would someone subscribe to Foundry instead?

If a feature does not strengthen that answer, redesign it or question whether it
belongs.

For every design and implementation decision, ask:

> **Does this make Foundry feel like an elite engineering team, or does it make
> it feel like another AI tool?**

## What Foundry Must Do Dramatically Better

### 1. Understand the whole project

Foundry understands more than the latest prompt. It builds a working model of
the codebase, architecture, history, previous missions, user preferences, and
established project patterns.

### 2. Plan before coding

Foundry behaves like a senior software architect. It discovers constraints,
forms a coherent plan, and anticipates consequences before editing code.

### 3. Execute autonomously

Foundry can carry large engineering missions through to completion with minimal
supervision while keeping the user appropriately informed and in control.

### 4. Verify continuously

Foundry checks builds, tests, lint, types, runtime behavior, previews, logic,
regressions, security, and performance. When safe, it diagnoses and fixes
problems automatically instead of merely reporting them.

### 5. Explain decisions naturally

Foundry communicates concise, professional engineering reasoning: what changed,
why it matters, what was verified, and what tradeoffs remain. It does not bury
the user in raw logs or expose private chain-of-thought.

### 6. Preserve continuity

Follow-up work feels like continuing with the same engineer. Project context,
decisions, preferences, progress, and unresolved concerns survive across
missions instead of resetting with every conversation.

### 7. Make software feel alive

Plans turn into real file edits, commands, running software, live previews, and
visible verification. Progress is grounded in evidence from the actual project.

### 8. Build beautiful software

Even without detailed design instructions, Foundry produces intentional,
coherent, accessible software rather than generic template output.

### 9. Learn each project

Foundry remembers useful decisions, learns recurring patterns, reuses knowledge,
and compounds trust without making the user repeat themselves.

### 10. Own the idea-to-production workflow

Discovery, architecture, implementation, testing, preview, deployment,
maintenance, and future improvement form one continuous workflow rather than a
collection of disconnected tools.

## The Experience We Are Building

Users should repeatedly feel:

- "This AI understands exactly what I'm trying to build."
- "I trust it."
- "It catches things I would have missed."
- "It keeps making my project better."
- "I don't want to build software without Foundry anymore."

These are experience requirements, not marketing copy. Product flows,
architecture, interface language, agent behavior, and engineering priorities
should make them true.

## Product Review Gate

Before accepting a meaningful feature or implementation, verify that it:

1. Strengthens at least one of the ten differentiators above.
2. Uses real project context and evidence wherever relevant.
3. Advances work rather than merely creating more conversation or ceremony.
4. Preserves user trust through clear control, verification, and honest status.
5. Fits the continuous idea-to-production relationship.
6. Feels intentional and integrated rather than like another chatbot, wizard,
   dashboard, or IDE feature.

When a proposal fails this gate, revise the experience before polishing the
implementation.
