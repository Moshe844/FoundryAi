## 1. Mission Architecture Map (as-built, verified 2026-07-12)

### Canonical state
- **Single store:** `WorkspaceState = { activeMissionId, missions: MissionState[] }` held in `components/WorkspaceShell.tsx` via `useState`. Mutated at **32 `setWorkspace(...)` sites**. Persisted to IndexedDB (source of truth) + localStorage (compact fallback) via `compactWorkspaceForLocalStorage` / `normalizeMission`.
- **Per-turn execution unit:** `ExecutionMission` (in `lib/mission-engine.ts`) with canonical `state: ExecutionMissionState` = idle | understanding | planning | executing | waiting_for_user | waiting_for_approval | verifying | complete | ... . A `MissionState` (conversation thread) holds an array of `ExecutionMission`s + `activeExecutionMissionId`.
- **Single derived-status selector:** `deriveMissionDisplayStatus(mission)` in `lib/mission/status.ts` → `{ state, label, isBusy, isPausedForUser, isPausedForApproval, activeExecutionMission }`. Derives ONLY from the active `ExecutionMission.state`. Consumers: `WorkspaceShell.tsx`, `BuildDashboard.tsx`. This is the intended single source of UI truth.
- **`getActiveExecutionMission(mission)`** in `status.ts` — the one resolver for "which ExecutionMission is active" (by id, else last).
- **State architecture stabilized (2026-07-13):** removed the unused `MissionContext`/reducer/translator path. `WorkspaceState` remains the sole persisted live store; `lib/mission/model.ts` now contains only the durable `ExecutionMission` contract and pure display selectors. There is no second imaginary mission store.

### Timers/loops (contradiction risk surface)
- WorkspaceShell: pendingWork step ticker (950ms), 150s stream-inactivity watchdog.
- BuildDashboard: connector health polls at 3s/3s/5s/10s (four separate intervals), preview polls.
- runtime.ts: server-side sweep interval (2368).

### Architectural verdict for Suite A
Single store + single selector already exist. Contradiction bugs, where present, come from **`setWorkspace` sites that mutate `liveWorkEvents`/`lastResult` without moving the active `ExecutionMission.state`**, so the selector reports stale state. Fixes should route all state moves through the ExecutionMission.state machine, not ad-hoc string fields.

## 3. Root Causes Found
- **RC-1 (PRE-1, fixed+verified):** Two independent intent classifiers with no mutation guard. The follow-up classifier returned `clarify` for concrete imperative change requests bundling contradictions; the client's `clarify` branch dead-ended with a passive note and retracted the mission, so `/api/factory/existing` never fired. Fix: deterministic imperative-mutation guard in `applyProjectIntentPolicy` → routes to `edit`; contradictions handled in-mission by the pausing DecisionPrompt.
- **RC-2 (PRE-2, fixed, UI retest pending):** the same `clarify` branch rendered ambiguity as a passive chat note instead of the inline one-at-a-time DecisionPrompt. Fix: `pendingClarification` on `MissionState` rendered via `DecisionPrompt`; intent route now emits `clarify_options`.
- **RC-A1 (Suite A, fixed):** `updateLiveChecklist` forced `state:"planning"` on every `checklistJson`-bearing event. Internal checklist-sync events (which skip the timeline re-derive) kept firing during executing/verifying, pinning the status pill to "Planning" mid-verification and flickering it on every non-internal checklist event. Fix: checklist carries plan data, not phase — only advance INTO planning from pre-plan states, never regress.
- **RC-D1 (Suite D, fixed):** Foundry (correctly) runs a read-only existence probe before installing a dependency, but the probe idiom `dir node_modules\pkg 2>nul || echo NOT_FOUND` was classified `unrecognized` → approval-required, so the user was prompted to approve a harmless directory listing and the mission stalled before the real `npm install`. Fix: `isReadOnlyExistenceProbe` recognizes read-only inspectors + stderr-suppression + `echo` fallback as safe, while still refusing real chains (`dir & del`, `; rm -rf`, `|| del`, `&& curl|sh`).
- **RC-D2 (Suite D, deny-loop — fixed):** Denial (and exact-command approval) matched the literal normalized command string, so denying `npm install dayjs --save` did NOT cover the model's follow-up `npm install dayjs` → the mission re-blocked (`awaiting-approval` again) instead of honoring the deny. The spec forbids deny loops and requires exact-command grants to ignore harmless formatting differences. Fix: `commandPermissionIdentity` canonicalizes the install alias (`i`→`install`) and save-family flags for JS package managers only; different packages, `--global`, `--force`, and other command families stay distinct (10/10 offline incl. safety cases). Wired into deny-match + exact-command approval/scope.
- **RC-D3 / architecture (found live, fixed, live-reverify blocked):** The local connector (`scripts/foundry-local-connector.cjs`) is a standalone process with **its own inline copies** of `decideCommandPermission`, `normalizeCommandText`, and the approval-bypass match — it does NOT import the app's TS. So the RC-D1 (probe) and RC-D2 (identity) fixes made in the TS lib applied only to the SERVER-access path (uploaded copies / browser-folder), NOT to the connector — which is the primary local-exec path and the one every live test here used. Deny still worked on the connector because deny is enforced server-side in the executor. approve-once exposed the gap (connector bypass used exact string match → `--save` variant re-blocked). Fix: ported both helpers into the connector, wired into `decideCommandPermission` + the bypass. **Honesty note:** the connector is a long-running process (PID 19376); the fix loads only on `npm run connector` restart, so connector-path approve-once/probe are fixed-but-not-yet-live-verified. Broader lesson for the audit: server-access and connector are two permission code paths that must be tested (and kept in sync) separately.
- **RC-D5 (Windows dependency-install failure — found live, fixed):** With approve-once working (the install variant was correctly allowed and RAN — RC-D2/D3 confirmed live), the install then failed: `npm.ps1 cannot be loaded because running scripts is disabled on this system` — PowerShell's execution policy (common default) blocks npm's `.ps1` shim. The connector's shell-fallback (`cmd → powershell → git-bash`) did NOT auto-recover because the execution-policy error wasn't in `shellMismatchPatterns`, so it surfaced a terminal failure; the model then retried via `cmd /c "npm install dayjs"` which re-prompted (wrapper changed the approval identity). Impact: dependency installs broken for any Windows user with a Restricted/RemoteSigned policy + PowerShell-sticky shell. Fix: added the execution-policy errors to `shellMismatchPatterns` so the connector auto-falls-back to cmd (which uses `npm.cmd`, no `.ps1`) and reuses the existing approval — no re-prompt. Verifying on a fresh repo connector (:3919).
- **RC-D4 (operational, found live):** The running "Local Agent" on :3917 is **not** the repo's `scripts/foundry-local-connector.cjs` — it's an **installed copy** at `%LOCALAPPDATA%\FoundryLocalAgent\foundry-local-connector.cjs`, deployed by `app/api/factory/agent/download/route.ts` and auto-started at login via a Startup VBS. So editing the repo file (or "restarting the connector") does NOT update the running agent — the installed copy was stale (0 of my fixes). This is why two "restart" cycles kept executing old code. Confirmed by diffing installed-vs-repo. **Two actions:** synced the fixed repo file into the install dir; and verified the fix live by launching a fresh repo connector on :3918 (`FOUNDRY_CONNECTOR_PORT=3918`) — the read-only probe that was blocked on :3917 now runs (exit 0). **Ledger note for release:** the download/install flow needs a version stamp + "update available / re-install" path, or the agent should self-update, so security-relevant connector fixes actually reach users. Ties to the deferred packaged-installer story.
- **RC-OBS-1 (tooling, not a product bug):** In-app browser `computer{screenshot}` times out (30s) on the Foundry workspace because 4 connector health-poll intervals (3s/3s/5s/10s) + a 950ms ticker keep the renderer from ever idling. `get_page_text` and DOM/ref tools work fine. Screenshot evidence for the ledger will come from a Playwright driver (own wait strategy) rather than the MCP screenshot action.

## 4. Refactors Performed
- Ported permission logic into the connector (`isReadOnlyExistenceProbe`, `commandPermissionIdentity`) so its inline classifier matches the TS lib. On-disk verified complete (probe→decideCommandPermission L340; identity→bypass L961). **Running connector still executes pre-edit code** (a live Node process doesn't hot-reload; empty `approvedRoots` was NOT a restart signal — my earlier inference was wrong). Requires `npm run connector` restart to load. **Open follow-up:** these two files (`command-permissions.ts` and the connector `.cjs`) are hand-synced duplicates — a shared generated source or a connector build step would prevent future drift.

## 5. Automated Tests Added
- **`scripts/eval-command-permissions.cjs`** (`npm run eval:command-permissions`) — compiles the real classifier + guards its behavior: baseline safe/permission-required/destructive classification; RC-D1 read-only existence probes (bare, shell-wrapped, piped-to-filter) allowed while adversarial chains (`dir & del`, `dir | del`, wrapped `Remove-Item`, `; Remove-Item`) refused; RC-D2 permission-identity collapses install-flag variants but keeps different packages / `--global` / other command families distinct. **26/26 passing.** Directly covers the spec's required regressions "approval deduplication" and "denial does not loop" at the classifier layer.
